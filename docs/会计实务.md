## 一、银行记账实务

### 1.1 直录（平时实时记账）

**场景**: 登录网银看到银行扣了手续费/给了利息，直接记一笔。

**会计实务做法**:
- 手续费: `dr 6603 财务费用 cr 1002 银行存款`（减少银行日记账余额）
- 利息收入: `dr 1002 银行存款 cr 6603 财务费用-利息收入`（增加银行日记账余额）

**系统对应**: `POST /api/bank/entry`
- 同时生成 BankTransaction（银行日记账流水）+ 会计凭证（AccountMove）
- 适用于平时见一笔记一笔的实时记账

**为什么这样做**: 小微企业会计实务中，出纳登录网银看到银行已扣款/已入账，立即更新银行日记账是最常见的操作方式。等到月末银行对账单来了再逐笔核对，已记的会自动匹配。

### 1.2 对账补录（期末统一处理）

**场景**: 期末对账时发现银行对账单上有手续费/利息，但系统里没记。

**会计实务做法**: 在调节表中列为"银行已付企业未付"项，生成补录凭证。

**系统对应**: reconciliation → `POST /api/bank/reconciliation/{id}/generate-entry`
- 只生成会计凭证（AccountMove），不产生 BankTransaction
- 适用于对账时发现的遗漏项

### 1.3 直录 vs 对账流程

| 维度 | 直录 | 对账流程 |
|------|------|----------|
| 记账时机 | 平时实时 | 期末统一 |
| 触发条件 | 登录网银看到 | 对账时发现 |
| 系统接口 | `POST /api/bank/entry` | `POST /api/bank/reconciliation/{id}/generate-entry` |
| 生成 BankTransaction | ✅ | ❌ 只生成凭证 |
| 生成会计凭证 | ✅ | ✅ |
| 是否需要之前导入对账单 | ❌ | ✅ |

## 二、往来款实务

### 2.1 销售收款

**会计实务**:
```
dr 1002 银行存款（实际收到）
cr 1122 应收账款（冲销应收）
```

**系统对应**: `POST /api/receipts` — 销售收款时带 `bank_account_id` 自动产生银行流水

**实务要点**:
- 创建销售单时若勾选"已收款"（`payment_status=paid`），系统自动生成收款记录和凭证
- 先开销售单后收款，用 `POST /api/receipts` 单独录收款
- 销售出库时结转成本: `dr 6401 主营业务成本 cr 1405 库存商品`

### 2.2 采购付款

**会计实务**:
```
dr 2202 应付账款（冲销应付）
cr 1002 银行存款（实际支付）
```

**系统对应**: `POST /api/payments` — 采购付款时带 `bank_account_id` 自动产生银行流水

### 2.3 冲销（红字更正）

**会计实务**: 收款/付款/发票/费用等录错时，不能用 DELETE 删除，用红字冲销：
- 生成反向分录（方向相反、金额为负）
- 原记录保留，审计痕迹完整

**系统对应**:
| 端点 | 用途 |
|------|------|
| `POST /api/receipts/{id}/reverse` | 红冲收款（自动生成反向 BankTransaction + 红字凭证） |
| `POST /api/payments/{id}/reverse` | 红冲付款（同上） |
| `POST /api/bank/transaction/{id}/reverse` | 红冲银行交易（同上） |
| `POST /api/invoices/{id}/reverse` | 红冲发票（标记原发票+创建红字发票+级联冲红凭证库存） |
| `POST /api/expenses/{id}/reverse` | 红冲费用（冲红总账凭证+标记 is_reversed） |
| `POST /api/cash-flows/transactions/{id}/reverse` | 红冲现金流水（冲红总账凭证+标记 is_reversed） |

**发票冲红级联规则**:
- 销项发票（关联销售单）→ 冲红收入/应收/税额凭证 + 库存回退
- 进项发票（关联采购单）→ 冲红存货/应付/税额凭证 + 库存退回
- 独立发票 → 仅标记+创建红字发票

**危险操作拦截**:
- 所有上述实体的 `DELETE` 被 `readonly_middleware.py` 拦截（403），强制走 reverse/cancel
- 销售单/采购单用 `POST /{id}/cancel` 取消（保留审计轨迹+冲红凭证库存），不可 DELETE

> 红冲是危险操作，`/reverse` 端点受 AI 白名单约束（receipts/payments/bank/transaction 三个已在白名单内，AI 可调；发票/费用/现金流 reverse 不在白名单，需用户前端操作）。

## 三、税费实务

### 3.1 增值税

**实务分类**:

| 纳税人类型 | 税率 | 分录模式 |
|-----------|------|---------|
| **一般纳税人**（销项-进项） | 13%/9%/6% | 发票自动生成 dr 应收账款 cr 主营业务收入 + 应交税费-销项 |
| **小规模纳税人**（简易征收） | 1%，季度总销售额≤30万时普票免征 | 同上（税率不同） |

**小规模普票/专票免税逻辑**（2026.1.1-2027.12.31 政策）:

```
季度总销售额（普票+专票合计）≤ 30万：
  普票收入 → 免征增值税，tax = 0
  专票收入 → 减按1%征收，tax = 专票收入 ÷ (1+1%) × 1%

季度总销售额 > 30万：
  普票收入 → 减按1%征收
  专票收入 → 减按1%征收

附加税 = 实际缴纳增值税 × 12% × 50%（小微企业减半）
免税部分不计附加税
```

**月末结转**（BR-11）:
- 销项 > 进项: `dr 222106 转出未交增值税 cr 222107 未交增值税`
- 留抵: 不做分录，自然体现在应交增值税借方余额

**实务要点**:
- 增值税是负债（代收代付），不是费用
- 缴纳增值税只是清负债，不计入损益

### 3.2 附加税

**计算基数**: 当期应交增值税（不是次月实缴数）

**费率**: 城建税 7% + 教育费附加 3% + 地方教育附加 2%（合计 12%，小微企业减半为 6%）

**分录**: 月末计提 `dr 税金及附加 cr 应交税费-附加税`

### 3.3 企业所得税

**计提时点**: 月末月结时

**小微企业优惠**: 应纳税额 = 利润 × 25%（税率）× 20%（优惠）= 利润 × 5%

**分录**: `dr 所得税费用 cr 应交税费-企业所得税`

**主体类型**: `Account.type` 决定是否缴纳：
- `company` → 正常计算企业所得税（5%/25%）
- `personal`（个体工商户）→ 不缴企业所得税，系统返回 0

**亏损处理**: 亏损时 `tax_payable=0`，不再抛异常

## 四、费用分类实务

### 4.1 费用类目归属

按《小企业会计准则》利润表结构：

```
营业收入 - 营业成本 = 毛利润
  - 销售费用(5601)
  - 管理费用(5602)
  - 财务费用(5603)
  - 税金及附加(5403)
= 营业利润
  ± 营业外收支
= 利润总额
  - 所得税费用(5801)
= 净利润
```

**系统使用企业会计准则科目编码（6601/6602/6603/6403/6801），但分类结构与上表一致。**

各 `category` 的会计准则归属：

| category | 会计准则归属 | 入账科目 | 说明 |
|----------|-------------|---------|------|
| `房租` | 管理费用 | 6601/5602 | 办公/仓库租金，属固定运营支出 |
| `水电` | 管理费用 | 6601/5602 | 日常运营水电 |
| `工资` | 管理费用/销售费用 | 6601/5602 或 6602/5601 | 管理部门入管理费用，销售部门入销售费用 |
| `办公用品` | 管理费用 | 6601/5602 | 文具/打印/日用品 |
| `材料` | 管理费用 | 6601/5602 | 仅限**非生产性零星耗材**；生产用原材料应走采购入库 |
| `运费` | 销售费用 / 存货成本 | 6602/5601 | **销售运费**入销售费用；**采购运费**应计入存货采购成本，不走费用 API |
| `维修` | 管理费用 | 6601/5602 | 日常维修维护 |
| `其他` | 管理费用 | 6601/5602 | 兜底 |
| `税金及附加` | **税金及附加** | 6403/5403 | 城建税/教育费附加/地方教育附加，月末计提 |
| `所得税` | **所得税费用** | 6801/5801 | 企业所得税，月末计提，**不是税金及附加** |

> **关键**：税金及附加(6403)与所得税费用(6801)是利润表两个独立行次。当前系统将所得税也映射到 6403（代码 bug，`EXPENSE_ACCOUNT_CODE_MAP` 缺少 6403 和 6801 条目），月末计提已通过 `engine_tax.py` 直接入账。日常通过 expense API 录所得税时，实际借记科目为 6601（默认回退），这是不正确的。

> **运费提示**：采购运费按《小企业会计准则》应计入存货采购成本（存货的组成部分），不应单独作为费用。只有销售运费才入销售费用。
>
> 银行手续费和利息收入**不走 expense API**，走 `POST /api/bank/entry`（见 §1 银行记账实务）。

**实务要点**: 利润表格式为 `营业收入 - 营业成本 - 税金及附加 - 销售费用 - 管理费用 - 财务费用 = 营业利润`

## 五、固定资产实务

### 5.1 新增资产

**入账价值**: 买价 + 运杂费 + 安装费（达到预定可使用状态前的一切合理支出）

**系统操作**: `POST /api/fixed-assets`

### 5.2 折旧

**起提时点**: 当月增加下月提（BR-12: 月结自动计提）

**方法**: 年限平均法（默认），可选双倍余额递减/年数总和法

### 5.3 处置

**系统流程**:
1. 先折旧到处置当月
2. 录入处置收入
3. 系统自动计算处置损益: `dr 银行存款（处置收入）cr 固定资产清理（净值）`，差额入资产处置损益

## 六、库存实务

### 6.1 成本流转

**计价方法**: 移动加权平均法（BR-7）

**采购入库**: `dr 1405 库存商品 cr 1002/2202 银行存款/应付账款`
- `unit_price` = 采购单价（用户录入）
- 更新 `Inventory.average_cost`（移动加权平均）

**销售出库**: `dr 6401 主营业务成本 cr 1405 库存商品`
- `unit_cost` = 出库时的 `average_cost`（锁定到 SaleItem）
- 禁止直接修改 SaleItem.unit_cost（`@property` 只读）

### 6.2 库存调整

> **必填原因**（BR-20）：任何库存调整（增加或减少）都必须填写 `reason` 字段，前端弹窗二次确认。

**盘盈（实物多账面少）**: `InventoryEngine.inbound(source_type="adjustment")`
- 补录入库，按当前平均成本入账

**盘亏（实物少账面多）**: `InventoryEngine.outbound(source_type="adjustment")`
- 出库减少库存，差额入管理费用

## 七、记账原则

### 7.1 财务数据不可变性

会计实务中，已入账的财务数据不得直接修改或删除。系统遵循此原则：

| 数据类型 | 更正方式 | 系统端点 |
|---------|----------|---------|
| 收款录错 | 红字冲销 | `POST /api/receipts/{id}/reverse` |
| 付款录错 | 红字冲销 | `POST /api/payments/{id}/reverse` |
| 银行交易录错 | 红字冲销 | `POST /api/bank/transaction/{id}/reverse` |
| 发票录错 | 红字冲销 | `POST /api/invoices/{id}/reverse` |
| 费用录错 | 红字冲销 | `POST /api/expenses/{id}/reverse` |
| 现金流水录错 | 红字冲销 | `POST /api/cash-flows/transactions/{id}/reverse` |
| 采购单录错 | 红冲后重建 | `POST /api/purchases/{id}/cancel` |
| 销售单录错 | 红冲后重建 | `POST /api/sales/{id}/cancel` |
| 凭证录错 | 冲红凭证 | 系统自动 |
| StockMove 已生成 | 库存调整单 | `InventoryEngine.reverse()` |
| 库存盘点 | 填写原因调整 | `PUT /api/inventory/{id}`（reason 必填） |

#### 红字冲销实例：利息录入错误

**场景**: 用 `POST /api/bank/entry` 录利息时 `entry_type` 传错（如用了 `"interest"` 而非 `"interest_income"`），利息被记为 outflow。

**错误结果**:
- BankTransaction: outflow 0.61（两笔合计）
- 会计凭证: 贷方 0.61（dr 6603 cr 1002，方向反了）
- 银行日记账余额减少 0.61（应为增加）

**补救（错误做法）**: 又用 `entry_type="interest_income"` 录入正确 inflow。结果错误 outflow + 正确 inflow 互相抵消，利息收入净额为零——看似"平的"但业务含义全错。

**补救（正确做法）** — 红字冲销：
1. 对错误 outflow 逐笔调 `POST .../reverse`（系统生成反向 BankTransaction + 红字凭证）
2. 正确 inflow 维持不变
3. 最终净效果: `inflow 0.61 + outflow(-0.61) + inflow 0.61 = inflow 0.61` ✓
4. 审计痕迹: 原错误 → 冲销 → 正确，三笔记录完整可追溯

**关键原则**: 错误的 outflow 不能留在系统里等正确 inflow 来"对冲"——两笔方向相反但都是独立合法的交易。只有通过冲销明确标记"此笔作废"，审计才能理解业务全貌。

### 7.2 银行流水真相

BankTransaction 必须是业务操作（收款/付款/直录）的副产品，不得直接 INSERT：

| 创建方式 | 说明 |
|---------|------|
| `POST /api/receipts` | 收款时自动生成 |
| `POST /api/payments` | 付款时自动生成 |
| `POST /api/bank/entry` | 利息/手续费直录时自动生成 |

> **期初余额不产生 BankTransaction**：`POST /api/opening-balances` 的 `bank_balance` 过账到总账 1002 科目余额，但不写入 BankTransaction 表。
> 
> **银行账户创建时 balance=0 是故意设计**：不通过手动设初始化余额，而是引导用户导入银行对账单（含期初余额）→ 执行对账 → 确认调节表，让银行余额来自银行流水而非手动输入。这样做确保 BankTransaction 只记录真实业务流水，不混入期初调整。老用户迁移时同样适用——导入截至今天的历史对账单，对账即完成期初确认。

## 八、实务决策树

用户说"银行扣了手续费"：

```
登录网银看到？
  ├── 是 → 直录（POST /api/bank/entry）
  │      分录: dr 6603 财务费用 cr 1002 银行存款
  │      结果: 银行日记账余额更新 + 会计凭证生成
  │
  └── 否（对账时才发现）→ 对账流程
          ├── 导入对账单（POST /api/bank/statement）
          ├── 执行对账（POST /api/bank/reconcile）
          ├── 生成凭证（POST /api/bank/reconciliation/{id}/generate-entry）
          └── 确认调节表（POST /api/bank/reconciliation/{id}/confirm）
          结果: 生成会计凭证，不产生银行流水（银行流水已在对账单上）
```
