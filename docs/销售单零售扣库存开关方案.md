# 销售单零售扣库存开关方案（deduct_inventory）

> 版本：v1.1 | 日期：2026-04-29 | 适用：SQLite | 状态：审查修正版

---

## 一、背景与目标

在现有“项目业务走领料（ProjectCost 材料类联动库存）”的前提下，引入销售单级别开关 `deduct_inventory`：

- **零售销售（deduct_inventory=true）**：销售单创建/删除/取消/恢复/改数量时，**直接扣减/回补库存**。
- **项目销售（project_id 非空）**：销售单**不扣库存**，库存出库继续统一走“领料/材料成本联动”。

> 关键约束：禁止出现“项目销售单 + 扣库存”导致的双扣。执行层应做到**默认不扣**、并在 project_id 有值时强制不扣或直接拒绝。

---

## 二、业务规则（强约束）

### 2.1 默认值与旧数据

- **默认值**：新建销售单 `deduct_inventory` 默认 **false**（最安全，避免项目单误扣）。
- **旧数据**：不回填该字段。历史记录若为 `NULL`，逻辑上按 **false** 处理（不追溯历史库存）。

### 2.2 约束关系

- 若 `project_id` **非空**：
  - `deduct_inventory` 必须为 **false**。
  - 推荐实现为：后端强制将其置为 false，或返回 400 并提示“项目单走领料，不支持销售单扣库存”。
- 若 `project_id` **为空**（零售）：
  - `deduct_inventory` 可为 true/false。
  - true：销售单承担出库凭证角色；false：销售单仅记账不出库（可用于先开单后出库/其他流程）。

### 2.3 影响库存的事件（仅当 deduct_inventory=true）

- **create_sale_order**：
  - 按 items 扣库存（逐行校验库存充足，失败需整体回滚）。
- **delete_sale_order**：
  - 回补库存（逐行回补）。
- **update_sale_order**：
  - 若 items 发生变更：按差值调整库存（可用“先回补旧、再扣减新”的模式保证可回滚）。
  - 若 status 从 completed→cancelled：回补库存；cancelled→completed：重新扣减库存。
- **幂等与并发**：
  - 同一销售单的库存调整必须与销售单、明细行、日志在同一事务中提交；任一步失败必须 rollback。

### 2.4 允许扣库存的前置条件（必须同时满足）

销售单触发扣库存必须同时满足：

1. `deduct_inventory == true`
2. `project_id IS NULL`
3. `status == 'completed'`（或系统定义的“生效”状态）

> 目的：避免“草稿/取消单也扣库存”、以及“项目单双扣”两类高频事故。

## 三、数据层改造（models + schemas + 迁移）

### 3.1 models.py（SaleOrder）

- 新增字段：`deduct_inventory`
  - SQLite 推荐存储为 `INTEGER`（0/1）或 `BOOLEAN`（实际仍是 0/1）。
- （可选）增加索引：`ix_sale_orders_deduct_inventory`（按需，不是必须）。

> 兼容性约定：读取时将 NULL 视为 0（false），避免旧数据分支污染业务判断。

### 3.2 schemas.py

- `SaleOrderCreate`：新增 `deduct_inventory: Optional[bool] = False`
- `SaleOrderUpdate`：新增 `deduct_inventory: Optional[bool] = None`（仅在需要允许修改时）
- `SaleOrderOut`：新增 `deduct_inventory: bool`（输出层必须归一化：NULL→false）

> 输出层建议做兼容：若数据库该列为 NULL，序列化时按 false 输出，避免前端出现 null 分支。

### 3.3 database.py 迁移（SQLite）

在 `_migrate_linkage(engine)` 或独立迁移函数中新增：

- `ALTER TABLE sale_orders ADD COLUMN deduct_inventory INTEGER DEFAULT 0`
- （可选）`CREATE INDEX IF NOT EXISTS ix_sale_orders_deduct_inventory ON sale_orders(deduct_inventory)`

> 旧数据列值为 NULL/0 都按 false 处理；本方案不追溯历史库存。

## 四、后端实现（推荐：联动逻辑下沉到 linkage）

### 4.1 linkage 需要新增的能力

在 `backend/crud/linkage.py` 中新增一组“销售单扣库存”函数（与项目领料的材料成本联动隔离）：

- `sale_deduct_inventory(db, account_id, order, operator)`
- `sale_restore_inventory(db, account_id, order, operator)`
- `sale_update_inventory(db, account_id, order, old_items_snapshot, operator)`

核心原则：

- **仅在 `order.deduct_inventory == True` 且 `order.project_id is None` 时执行**。
- 逐行校验库存充足；失败抛出异常，外层统一 rollback。
- 写入 `_log`（必须是 flush，不在 _log 内 commit）。

> items 快照：建议在 update 时先读取旧明细行（product_id、quantity），再对比新明细行，按差值调整库存；或采用“先回补旧、再扣减新”以简化一致性。

#### 4.1.1 推荐的库存调整算法（语义更清晰）

- **更新 items**：优先用“差值法”：
  - 对每个 product_id 计算 `delta = new_qty - old_qty`
  - `delta > 0`：追加扣减；`delta < 0`：回补 `abs(delta)`
  - 对 `delta > 0` 的项先校验库存充足，再统一扣减，避免中途失败造成难读的日志顺序
- **也可用回补-再扣减**（实现更简单，但会产生两段日志）：必须依赖事务 rollback 保证原子性。

### 4.2 orders.py（事务编排）

在 `create_sale_order/update_sale_order/delete_sale_order` 中按以下顺序组织事务：

```text
db.add(order) / 更新字段
db.flush()  (拿到 order.id / 确保明细行可用)
若 deduct_inventory=true 且 project_id is NULL 且 status 生效：调用 sale_*_inventory 扣/回补
若关联项目：仍按既有方案生成/删除 ProjectIncome（sale_create_income/sale_delete_income）
若影响到项目利润：调用 update_project_summary（在 commit 前）
db.commit()
异常：db.rollback()
```

### 4.3 防双扣建议（必须落地其一）

- **强制策略（推荐）**：若 `project_id` 非空，后端强制 `deduct_inventory=false`（忽略前端传值），并在响应中返回实际值。
- **拒绝策略**：若 `project_id` 非空且 `deduct_inventory=true`，返回 400，提示“项目单走领料，不支持销售单扣库存”。

> 本方案推荐使用“拒绝策略”，因为强制策略会让前端误以为已扣库存（若前端未显示回写值），而拒绝策略能在测试期更快暴露调用方错误。

## 五、前端交互（Sales.vue）

### 5.1 新增开关

- 新增开关字段：`deduct_inventory`
  - 文案示例：`零售出库（销售单直接扣库存）`
  - 默认关闭（false）

### 5.2 与项目选择联动

- 当用户选择了 `project_id`：
  - 自动将 `deduct_inventory` 置为 false
  - 禁用该开关并提示：`项目业务库存走领料，不在销售单扣库存`

### 5.3 与状态联动（避免取消单扣库存）

- 仅当销售单状态为“已完成/生效”时，允许开启 `deduct_inventory=true`
- 若状态切为 cancelled：后端触发回补；前端可在 UI 上提示“取消将自动回补库存”

## 六、测试用例（执行 AI 必跑）

### 6.1 零售扣库存（deduct_inventory=true, project_id=NULL）

- 创建：库存减少（每个商品 quantity 精确扣减）
- 删除：库存回补（数量一致）
- 更新数量：按差值调整库存（+/-）
- 取消/恢复：取消回补、恢复再扣（幂等、事务一致）
- 库存不足：接口返回 400，且订单/明细/库存均不落库（rollback 生效）

补充边界：

- 单据为 cancelled 时创建/更新：不应触发扣库存（由 2.4 前置条件约束）
- 同一销售单重复点击保存/网络重试：不应出现重复扣减（update 需按差值/幂等处理）

### 6.2 项目销售不扣库存（project_id!=NULL）

- 创建销售单：库存不变；收入自动生成（若方案启用收入联动）
- 传入 deduct_inventory=true：按“强制/拒绝”策略验证不会发生库存扣减

### 6.3 旧数据兼容

- 旧销售单 deduct_inventory 为空：不触发库存逻辑；查询/列表输出 deduct_inventory=false（或 null→false 显示）

---

## 七、回退策略

- 新增列可保留不删：回退时后端忽略 `deduct_inventory` 字段即可。
- 前端可隐藏该开关，恢复原行为（但建议仍保留项目单不扣库存的主口径）。
