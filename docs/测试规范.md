## Golden Tests: 独立会计师验证体系

> 核心理念：同一堆原始凭证，换一个独立会计师也能算出同样的报表。系统输出必须等于手工验算值。

### 方法论

每张 golden test 表分为三层：
- **L1 原始凭证** — 外部事实（数量、单价、税率），不依赖系统
- **L2 手工验算** — 按《小企业会计准则》逐条手算期望值，标注 §条文
- **L3 系统对照** — 创建业务后读回 StockMove/AccountMove/Ledger，与 L2 对照

### 核心原则（不得违反）

#### G-1: L2 期望值仅源自会计准则，严禁适应系统行为

L2 手工验算是**从《小企业会计准则》条文直接推导**的值，不是"系统当前输出值"。

```python
# ❌ 错误：围绕系统行为写期望
# 系统行为: enable_vat_deduction=False 时不计算税额 → 所以期望 0
OUTPUT_TAX = Decimal("0")

# ✅ 正确：从会计准则推导
# §5.1 第五十九条：小规模纳税人按征收率计税，但系统采用简易处理全额入账
# 此处按会计准则 = revenue * 3% = 30，系统输出不符则 FAIL
OUTPUT_TAX = (REVENUE * VAT_SMALL_SCALE_RATE).quantize(Decimal("0.01"))
```

**红牌场景**：
- 注释中出现"系统行为：" → 说明测试在解释系统的非标准行为
- 期望值因系统 bug 或限制而调整 → 应改为断言系统不符合准则

#### G-2: 已知系统缺陷不得被包容，应断言失败

发现系统已知缺陷（如冲红付款不被 CF 计入、BS 与 CF 不一致），golden test **必须断言失败**，不能降级为 warn 或调整期望值匹配。

```python
# ❌ 错误：用 warn 容忍不一致
CF_END = BANK_OPENING  # (≠BS 9850)
if abs(cf_end - bs_bank) > Decimal("0.05"):
    print("[WARN] CF期末≠BS银行存款")

# ✅ 正确：直接断言，发现即报
assert abs(cf_end - Decimal(str(bs["monetary_funds"]))) <= Decimal("0.05"), \
    f"CF期末({cf_end})≠BS货币资金({bs['monetary_funds']})"
```

**例外**：缺陷被明确标记为业务方确认的故意设计（BR-x），且 BR 文档有记录。即便如此，应在注释中注明 BR 编号而非"系统行为"。

#### G-3: 策略常量硬编码，不从系统导入

税率、所得税率等会计准则法定值必须**硬编码并标注 §条文**，不从 `policy/*.py` 导入。防止系统改错常量后 golden test 同步失效。

```python
# ❌ 错误：从系统导入
from policy.vat_facts import VAT_GENERAL_DEFAULT_RATE
TAX = VAT_GENERAL_DEFAULT_RATE.value  # 若系统改为15%，golden test 仍跟着绿

# ✅ 正确：硬编码 + 条文标注
# §1.3 第六十一条：一般纳税人销售货物适用 13% 税率
TAX = Decimal("0.13")

# 如果某项税率需要与系统一致（如小规模 3%→1% 阶段性政策），
# 用常量名注释说明政策文件依据
TAX_SMALL_SCALE = Decimal("0.01")  # 财政部 税务总局公告 2023年第1号：减按1%
```

#### G-4: 测试不得仅"不崩溃" —— 必须有数字级断言

所有 golden test 必须对具体科目余额、报表数字做值断言，不可仅验证 HTTP 200 或"不抛异常"。

```python
# ❌ 错误：仅验证不崩溃
r = c.post("/api/expenses", json={...})
assert r.status_code == 200

# ✅ 正确：验证具体科目
r = c.post("/api/expenses", json={...})
assert r.status_code == 200
db = _db()
try:
    assert _ledger_balance(db, "6601") == EXPENSE_AMOUNT
    assert _credit_balance(db, "2202") == EXPENSE_AMOUNT
finally:
    db.close()
```

**例外场景**：`test_golden_007_multicurrency` 中多币种创建被系统拒绝（422）可以接受——前提是系统**确实不实现该功能**且有 BR 文档记录。

#### G-5: 全量不变量每表必验

每张 golden test 表末尾必须执行：
- `AS-01` 全局借贷平衡
- BS 恒等式 `A = L + E`
- 追溯验证（`_trace_bal` 从报表到凭证行）

### 设计原则

1. **数字好算**：全部用 10 的倍数（100, 10, 1000）
2. **§条文标注**：每条期望值后标注准则条文，如 `# §31: 月折旧 = 原值×(1-残值率)÷月数`
3. **AS规则校验**：每个场景调用对应的 AS 规则（AS-01~AS-15）进行运行时校验
4. **StockMove/Inventory 双重验证**：所有涉及库存的场景同时验证 StockMove（L3 真相源）和 Inventory（L4 缓存）
5. **先退后付**：退货发生在付款前，避免计算混乱
6. **固资上月创建**：满足"当月增加下月提"规则
7. **共享基础设施**：`tests/golden/conftest.py` 统一 sys.path 和 client fixture；`tests/golden/helpers.py` 提取公共辅助函数
8. **双重验证**：每表同时做（a）逐科目余额断言 和（b）BS/IS 报表值断言，两条路径互相对照
