## 二、BUG 清单

### BUG-1 (P0 严重): 发票 auto_create 销售单/采购单 tax_rate 默认 1% 而非发票税率

**现象**：一般纳税人账本，发票指定 13% 税率，但生成的凭证中增值税按 1% 计算。

**证据**（试算平衡 2026-06-30）：

| 科目 | 预期 | 实际 | 差异 |
|------|------|------|------|
| 222101 销项税额(贷) | 25,740 | 2,060 | -23,680 |
| 222102 进项税额(借) | 26,000 | 2,000 | -24,000 |
| 1122 应收账款(借) | 232,780 | 208,060 | -24,720 |
| 2202 应付账款(贷) | 319,000 | 295,000 | -24,000 |

**计算验证**：206,000 × 1% = 2,060（销项），200,000 × 1% = 2,000（进项）→ 确认按 1% 入账。

**根因**：

| 文件 | 行号 | 问题代码 |
|------|------|---------|
| `backend/schemas/invoice.py` | 80 | `tax_rate: Decimal = Field(default=Decimal('0.01'), ...)` |
| `backend/commands/invoice_commands.py` | 847 | `tax_rate=it.get('tax_rate', invoice.tax_rate)` |
| `backend/commands/invoice_commands.py` | 933 | `tax_rate=it.get('tax_rate', invoice.tax_rate)` |

**数据流分析**：
1. API 调用方在 `items` 中不传 `tax_rate`（符合 SKILL.md 示例）
2. Pydantic 在 schema 校验阶段自动填入默认值 `Decimal('0.01')`
3. `it.get('tax_rate', invoice.tax_rate)` 中 `it.get` 总能取到 0.01，**回退到 `invoice.tax_rate` 的分支永远不会执行**
4. SaleItem/PurchaseItem 的 tax_rate 被设为 0.01，FinanceEngine.record_sale 按此税率计算销项税

**影响范围**：
- 所有通过 `POST /api/invoices/quick` + `auto_create` 创建的一般纳税人销售/采购业务
- 销项税、进项税、应收账款、应付账款全部按 1% 计算（应为 13%）
- VAT、附加税、所得税连锁错误
- 资产负债表不平衡

**修复建议**：
```python
# 方案A：schema 默认值改为 None（推荐）
# schemas/invoice.py line 80
tax_rate: Optional[Decimal] = Field(default=None, ge=0, le=1)

# 方案B：handler 中强制用 invoice.tax_rate 覆盖
# invoice_commands.py line 847/933
tax_rate=invoice.tax_rate,  # auto_create 时始终用发票税率
```

### BUG-2 (P1): 工资费用科目判断用拼音 "gongzi" 而非中文 "工资"

**现象**：工资计提凭证贷方记入 2202 应付账款，应记 2211 应付职工薪酬。发放时借 2211，导致 2211 出现借方余额（应为贷方）。

**证据**（试算平衡 2026-06-30）：

| 科目 | 预期 | 实际 | 问题 |
|------|------|------|------|
| 2211 应付职工薪酬 | 贷 5,000 | 借 25,000 | 方向相反，差额30,000 |
| 2202 应付账款 | 贷 289,000 | 贷 295,000 | 多出6,000（含工资30,000-24,000已付抵消） |

**根因**：

| 文件 | 行号 | 问题代码 |
|------|------|---------|
| `backend/routers/expenses.py` | 97 | `credit_code = "2241" if ... else "2211" if db_expense.category == "gongzi" else "2202"` |

实际 `db_expense.category` 的值是 `"工资"`（中文），永远不等于 `"gongzi"`（拼音），导致工资费用贷方落入 else 分支记为 2202。

而发放工资时（`backend/routers/payments.py` line 148-149）正确判断了 `data.payment_type == "salary"` 并借记 2211，造成借贷科目不对称。

**影响**：
- 2211 应付职工薪酬科目余额方向错误（借方而非贷方）
- 2202 应付账款虚高
- 资产负债表负债结构失真

**修复建议**：
```python
# expenses.py line 97
credit_code = "2241" if db_expense.payment_method == "private_advance" else "2211" if db_expense.category == "工资" else "2202"
```

### BUG-3 (P1): 期初银行余额不同步到银行账户记录

**现象**：期初余额录入 bank_balance=500,000 记入 1002 总账科目，但 bank_account.balance 保持 0。付款时系统检查 bank_account.balance 导致余额不足。

**证据**：
- 1002 银行存款（总账）余额：601,820（含期初50万 + 收款 - 付款）
- bank_account.balance：76,770（仅含通过银行账户的收付款：0 + 101,700 - 25,000 - 50 + 120）
- 付款 113,000 被拒："银行账户余额不足: 当前余额 76,770.00"

**根因**：期初余额（`POST /api/opening-balances`）的 `bank_balance` 直接生成 1002 总账凭证，但不更新 `BankAccount.balance`。两者独立维护，期初数据未同步。

**影响**：
- 有期初银行余额的公司，首月付款可能被错误拒绝
- 银行对账时账面余额与银行账户余额不一致
- 需要人工干预才能完成正常付款

**修复建议**：期初余额录入时，若存在银行账户，同步更新 bank_account.balance；或在付款余额检查时使用总账 1002 余额而非 bank_account.balance。

### BUG-4 (P2): 库存 API 不返回 average_cost 字段

**现象**：`GET /api/inventory` 返回的库存明细中不包含 `average_cost`（移动加权平均成本）字段，无法通过 API 查询库存均价。

**证据**：API 返回的库存项字段列表：
```
id, product_id, quantity, product_name, product_sku, product_category,
product_unit, min_stock, purchase_price, sale_price, last_updated, is_alert
```
缺少 `average_cost` / `total_value` 字段。

**影响**：SKILL.md §0 铁律1 要求"库存值读 Inventory.total_value 或 StockMove 汇总"，但 API 不暴露这些字段，只能通过试算平衡反推或直查数据库。

**修复建议**：在 `/api/inventory` 响应中增加 `average_cost` 和 `total_value` 字段。

---

### BUG-5 (P2): 危险操作确认流程返回值误导

**现象**：销售退货等危险操作（/return, /reverse, /cancel, /dispose）通过 202 确认后，返回的 JSON 中 `ok=false`，但实际操作已成功执行。

**证据**：
- 调用 `POST /api/sales/3/return` → 返回 202 + confirm_token
- 自动调用 `POST /api/confirm/{token}` 确认 → 操作成功（库存回补、凭证生成）
- 但脚本读到 `ok=false`，误判为失败

**影响**：API 调用者（包括 AI Agent）可能误认为操作失败并重复执行，或遗漏后续步骤。

**修复建议**：确认后的返回值应包含 `ok: true` 和操作结果，与直接执行一致。

## 三、BS 不平衡分析

月结前 BS diff = -26,940（资产 < 负债+权益），由以下 bug 叠加导致：

| 原因 | 对 BS diff 的影响 |
|------|------------------|
| BUG-1: VAT 按 1% 入账（销项少 23,680 + 进项少 24,000） | 应收少 24,720，应付少 24,000，VAT负债少 720 → 净影响 -26,940 中约 720 |
| BUG-2: 工资 2211 方向错误 | 2211 借方多 25,000（应贷 5,000），2202 多 30,000 → 净影响约 -5,000 |
| BUG-3: 付款失败 | 应付未冲减 113,000，银行未减少 113,000 → 净影响 0（两边都未动） |
| 退货确认后 ok=false 误导 | 退货实际成功，但首次脚本未正确处理 → 无实际影响 |

> 注：因月结被 BUG-3 连锁阻断（银行对账无法确认），月结未完整执行，所得税未计提（BUG-1 导致利润计算基数错误也有影响）。

## 六、结论

本次 6 月全场景测试覆盖了 13 类财务操作，发现 **5 个 bug**（1 个 P0 + 2 个 P1 + 2 个 P2）。

**最严重问题**是 BUG-1：发票 auto_create 时商品行项 tax_rate 默认为 1%（小规模税率），导致一般纳税人所有通过发票创建的采购/销售业务增值税全部按 1% 入账。这与昨天修复的 #10（engine_finance.py 移除 3% 覆盖）有关联但根因不同——#10 修复了 record_sale 中的税率覆盖逻辑，但 auto_create 在更上游就已经把 tax_rate 设错了。

建议优先修复 BUG-1 和 BUG-2，这两个修复简单但影响面广。
