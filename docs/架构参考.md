### 核心机制

| 机制 | 文件 | 用途 |
|------|------|------|
| Command Pattern | `commands/base.py` | 封装所有写操作 |

| BusinessError | `errors.py` | 统一业务错误处理 |
| AccountingEngine | `accounting_engine.py` | 会计准则统一计算引擎 + 输入/输出校验 |
| BankEngine | `engine_bank.py` | 银行引擎：BankAccount.balance 唯一更新入口 |
| FinanceEngine | `engine_finance.py` | 财务引擎：采购/销售会计凭证生成 + 校验 |
| FixedAssetEngine | `engine_fixed_asset.py` | 固定资产引擎：折旧计提/处置的唯一入口 |
| IntangibleAssetEngine | `engine_intangible_asset.py` | 无形资产引擎：摊销计算 |
| CostEngine | `cost_engine.py` | 成本引擎：加权平均 L2 计算真相源 |
| InventoryEngine | `engine_inventory.py` | 库存引擎：StockMove 流水唯一入口 |
| JournalEngine | `engine_journal.py` | 凭证引擎：5 种凭证模板 |
| LedgerEngine | `engine_ledger.py` | 科目余额管理、试算平衡 |
| ReceivableEngine | `engine_receivable.py` | 往来核销、账龄分析 |
| PeriodCloseEngine | `engine_period_close.py` | 期间结转（损益→本年利润） |
| BankReconcileEngine | `engine_bank_reconcile.py` | 银行对账引擎：对账单匹配、差异分析 |
| TaxEngine | `engine_tax.py` | 税务引擎：增值税/所得税计算 + 申报表 |
| TaxCheckEngine | `engine_tax_check.py` | 税务自查引擎：进项/销项匹配校验 |
| FinanceIntegration | `finance_integration.py` | JournalEngine 封装层 |
| ReadonlyMiddleware | `middleware/readonly_middleware.py` | 只读中间件：阻止直接修改关键数据 |
| ConfirmMiddleware | `middleware/confirm_middleware.py` | 确认中间件：红冲等不可逆操作二次确认 |
| AIGateway | `ai_gateway.py` | AI 网关：白名单校验 + AI 响应统一包装 |
| Auth | `routers/auth.py` | 登录认证：`POST /api/auth/login`，token 有效期 24h |
| AuthMiddleware | `account_dep.py:get_operator()` | 操作者识别：优先读 token 用户名，次读 `X-Operator` |
| ORM 防护 (before_update) | `models.py` / `models_finance.py` | `StockMove` / `FixedAssetDepreciation` / `AccountMove` 禁止 UPDATE |

## 🚫 设计约束

### 库存创建路径（单一入口）

库存记录只能通过以下两种途径创建，禁止直接写入 `Inventory` 表：

| 途径 | 触发操作 | 库存方向 |
|------|----------|----------|
| **采购入库** | 创建采购单（`POST /api/purchases`） | 增加 |
| **期初余额** | 创建期初余额时录入库存（`POST /api/opening-balances`） | 增加 |
| **库存调整** | 库存盘点调整（`POST /api/products/{id}/adjust-inventory`） | ± |
| **销售退货** | 销售退货（`POST /api/sales/{id}/return`） | 增加 |
| **采购退货** | 采购退货（`POST /api/purchases/{id}/return`） | 减少 |

**依据**：所有库存变动同时写入 `StockMove` 流水表（库存真相源），`Inventory` 表仅作为 L4 性能缓存。`StockMove` 一旦生成严禁修改删除，错误修正通过红冲调整单实现。

### 会计凭证保护

`AccountMove` 和 `FixedAssetDepreciation` 受 ORM 层 `before_update` 事件 + SQLite BEFORE UPDATE 触发器双重保护，禁止 UPDATE。

## 🔐 安全机制

### 多账本隔离

所有数据通过 `account_id` 隔离，API 请求必须携带 `X-Account-ID` 头。

```python
# 每个查询都必须过滤 account_id
items = db.query(Model).filter(Model.account_id == account_id).all()
```

### 操作审计

所有写操作记录到 `OperationLog` 表，包含：
- 操作类型 (create/update/delete/adjust/reverse)
- 实体类型和 ID
- 操作描述
- 操作者 (user/ai)

### 输入校验

- Pydantic Schema 校验请求格式
- BusinessError 校验业务规则
- Domain 层校验业务逻辑（库存充足、订单状态、科目合规性）

### 保护层

| 保护层 | 机制 | 拦截范围 |
|--------|------|----------|
| ReadonlyMiddleware | 403 拦截 | DELETE 发票/固定资产/费用/订单/期初余额等 |
| ConfirmMiddleware | 202 + confirm_token | POST reverse/cancel/dispose 等不可逆操作 |
| ORM before_update | 数据库层禁止 UPDATE | StockMove / AccountMove / FixedAssetDepreciation |
| SQLite BEFORE UPDATE 触发器 | 数据库层禁止 UPDATE | 同上，双重防护 |
| AI 网关白名单 | 403 + suggested_endpoint | AI Agent 写操作白名单校验 |

## 📊 性能考虑

### SQLite 优化

- 使用 WAL 模式提升并发读性能
- 合理使用索引
- 避免长时间锁表

### 查询优化

- 分页查询 (skip/limit)
- 按需加载字段
- 使用 joinedload 预加载关联
