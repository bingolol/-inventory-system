## 工作规范

### 任务启动

| # | 原则 | 说明 |
|---|------|------|
| S1 | **匹配 skill** | 接到任务后先 check 可用 skills 是否有对应专项（如 tdd/diagnose/review），有则加载使用 |
| S2 | **先读文档再动手** | 改代码前先读 CONTEXT.md 业务规则 + 开发速查表反模式红线，避免"修复"故意设计 |
| S3 | **计划先行** | 涉及 3+ 文件改动时，先列出计划让用户确认，不直接改 |
| S4 | **先查后建** | 建新文件前先 grep 是否已存在同类实现，不重复造轮子 |

### 代码修改

| # | 原则 | 说明 |
|---|------|------|
| S5 | **Command 写、CRUD 查** | 所有写操作必须走 Commands 模式，禁止在 Router 或 CRUD 中直接写数据库 |
| S6 | **业务规则不改** | CONTEXT.md 中标记为"故意设计"的规则（BR-1~BR-6）不得"修复" |
| S7 | **枚举单一源** | 新增枚举值只改 `backend/enums.py`，前端从 enumsStore 获取，禁止硬编码 |
| S8 | **金额用 Decimal** | 金额字段全部使用 `Decimal` + `round(2)`，禁止 `float` |
| S9 | **错误用 BusinessError** | 业务异常统一抛 `BusinessError(ErrorCode.XXX, message)`，不抛裸 Exception |

### 自检清单

| # | 检查项 | 说明 |
|---|--------|------|
| C1 | 测试通过 | `pytest` 全量通过后再提交 |
| C2 | 无旧代码残留 | 被替代的旧函数/字段/端点必须删除 |
| C3 | 文档同步 | 新增模块后检查 docs/ 下对应文档是否需要更新 |
| C4 | 无死 imports | 编辑过的文件检查 import 是否有多余 |

### Agent 沟通

| # | 原则 | 说明 |
|---|------|------|
| G1 | 不确定就问 | 业务规则不清楚、涉及 DB 结构变更、跨模块重构时，先问用户再动 |
| G2 | 不编造数据 | 调用 API 必须用真实响应，禁止假设 ID/金额/数量 |
| G3 | 不自作主张 | 不主动 commit/push，除非用户明确要求 |

## 快速导航

### 开发流程

```
理解任务 → 匹配 skill → 读文档 → 制定计划 → 执行 → 自检
```

### 关键约束

- **测试**: `pytest`（单元 + 集成 + E2E）
- **错误处理**: 使用 `BusinessError`（见 `backend/errors.py`）
- **金额精度**: `Decimal` + `round(2)`
- **写操作**: 必须通过 Commands 模式（`backend/commands/`）
- **枚举**: 单一真相源 `backend/enums.py`
