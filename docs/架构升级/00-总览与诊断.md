## ⚠️ 强制工作流程（每个阶段必须严格遵守）

```
1. 写方案    → 从架构视角出发，输出简洁优雅的代码方案（涉及哪些文件、改什么、怎么改）
2. 代码审查  → 语法检查 + 逻辑检查 + 架构检查，确保方案无遗漏无错误
3. 修正方案  → 根据审查结果修正方案
4. 再次审查  → 修正后的方案再次审查语法+逻辑+架构，确认无问题
5. 执行修改  → 按修正后的方案改代码
6. 测试验证  → 对更新过的功能和模块进行实际测试，确保100%落地无bug
7. 通过后更新文档进入下一个阶段
```

**禁止**：不写方案直接改代码、改完不测试就跳到下一个

## 一、现状诊断：为什么"牵一发而动全身"

### 1.1 六大深层根因

| # | 根因 | 典型表现 | 后果 |
|---|------|----------|------|
| R1 | **Float存储金额** | `finance.py:244` 用 `abs(diff)>0.01` 容差比较 | 浮点累积误差导致财务报表不平，修复一个数字引发更多不一致 |
| R2 | **隐式副作用链** | `create_sale_order()` 隐式触发库存扣减→收入创建→汇总重算→日志写入 | 改一行不知道影响什么，无法跳过/禁用/单独测试 |
| R3 | **双commit破坏原子性** | `project_costs.py:73` commit后，`:85` 又commit日志 | 两次commit之间崩溃=半残数据，无法回滚 |
| R4 | **ORM模型泄漏到业务层** | 路由层直接操作 `models.SaleOrder`，改字段波及6个文件 | 改一个字段名需改models/schemas/crud/routers/前端，变更面失控 |
| R5 | **枚举字符串散落前后端** | `payment_status` 的 `"paid"/"unpaid"/"pending"/"partial"` 到处硬编码 | 新增状态漏改一处就是bug，前后端枚举不同步 |
| R6 | **前端巨型组件** | `Projects.vue` 39KB/900+行，7个逻辑域挤在一处 | 改成本逻辑误触收入逻辑，diff难以审查 |

### 1.2 v1.0方案的不足（为什么需要v2.0）

v1.0方案（P0-P5六个阶段）解决了表层问题，但存在5个深层缺陷：

| # | v1.0做法 | 问题 | v2.0改进 |
|---|----------|------|----------|
| D1 | Repository返回dict | 丢失类型安全，Pydantic `from_attributes=True` 本就可处理ORM对象，手动序列化是重复劳动 | Repository返回TypedDict/ORM对象，Service层负责转换 |
| D2 | 领域事件同步执行 | handler异常中断后续handler，缺少优先级/过滤/条件触发 | 引入事件中间件链 + 条件过滤 + 优先级排序 |
| D3 | Service直接操作ORM模型 | 本质只是把crud代码搬到service，没有真正的领域建模 | 引入领域模型（Domain Model），Service操作领域对象而非ORM |
| D4 | schemas.py未拆分 | 686行全挤在一处，新增schema找不到位置 | 按领域拆分到 `schemas/` 目录 |
| D5 | 前端只有composable | 缺少状态管理（Pinia）、类型安全、API响应校验 | 引入Pinia store + 响应式缓存 + Zod校验 |

## 二、升级目标

### 2.1 核心目标

1. **改一个字段只改2个文件**（models + repositories），而不是6个文件
2. **每个业务操作只有1个commit点**，任一步骤失败全部回滚
3. **金额计算精确到分**，消除浮点累积误差
4. **新增联动逻辑只加1个函数**（`@on("event_name")`），不改现有代码
5. **枚举单一真相源**，后端定义、前端自动同步
6. **前端组件<200行**，逻辑域独立可测
7. **新增业务操作只加1个Command类**，不改路由层逻辑
8. **不变量自动校验**，每个写操作后自动触发验证

### 2.2 不变量保证（延续现有三大不变量）

| # | 不变量 | 含义 |
|---|--------|------|
| I | 库存不变量 | 材料成本变动后库存变化=数量差值，失败自动回滚 |
| II | 收入不变量 | 同一SaleOrder最多一条source_type='sale_order'的ProjectIncome |
| III | 汇总不变量 | projects.total_income/total_cost/profit永远可由明细重算得到 |

## 四、最终架构总览

### 4.1 后端分层

```
Router层（薄层，<50行/路由）
  │  只做：参数校验 → 构建Command → dispatch → 构建响应
  │  不做：任何业务逻辑、ORM操作
  ▼
Command层（业务操作=Command类）
  │  只做：加载Domain → 调用Domain方法 → 持久化 → emit事件
  │  不做：直接SQL、跨领域逻辑
  ▼
Domain层（业务规则+不变量）
  │  只做：状态机、规则校验、金额计算、validate()
  │  不做：数据库操作、事件触发
  ▼
Repository层（数据访问+ORM↔Domain转换）
  │  只做：ORM查询、Domain.from_orm()、update_from_domain()
  │  不做：业务逻辑
  ▼
EventBus层（副作用分发）
  │  只做：handler注册、中间件链执行、条件过滤
  │  不做：业务逻辑
  ▼
Models层（ORM模型定义）
```

### 4.2 数据流

```
用户请求
  → Router：构建 CreateSaleOrderCommand
    → dispatch(cmd, db)
      → with unit_of_work(db):
          → Handler：SaleOrderRepo.get_domain() / save_new()
          → Domain：domain.transition_to("completed") / domain.validate()
          → Repo：update_from_domain(db, domain)
          → emit("sale_order.created", ...)
            → Middleware：logging → handlers → invariant_check
      → commit（唯一commit点）
  → Router：构建响应返回
```

### 4.3 不变量保证体系

| 层级 | 机制 | 保证 |
|------|------|------|
| Domain层 | `domain.validate()` | 每个Domain对象自校验 |
| Command层 | handler内调用validate() | 业务操作前强制校验 |
| EventBus中间件 | `invariant_check_middleware` | 写操作后自动全局校验 |
| 数据库层 | UNIQUE索引 | 最终防线（如uq_income_source） |
| API层 | `/api/projects/verify-invariants` | 手动触发全量校验 |
