## 1. 高层架构概述

进销存管理系统是一套面向中小企业的全栈业务管理平台，涵盖**库存管理、采购/销售、项目归集、财务报表及个人账单**等核心模块。

### 系统架构图

```
┌─────────────────────────────────────────────────────────────┐
│                      浏览器 / 客户端                          │
│  ┌──────────────────────────────────────────────────────┐   │
│  │           Vue.js 3 SPA  (Vite + Pinia)               │   │
│  │  Views · Components · Composables · Stores · Router   │   │
│  └──────────────────────┬───────────────────────────────┘   │
└─────────────────────────┼───────────────────────────────────┘
          HTTP REST API   │  X-Account-ID Header               
          (Axios)         │  Bearer Auth                        
┌─────────────────────────▼───────────────────────────────────┐
│                    FastAPI Backend                           │
│  Routers → Commands → CRUD / Domain → Events → EventBus     │
│  Middleware · Exception Handlers · Static Files             │
└─────────────────────────┬───────────────────────────────────┘
                          │ SQLAlchemy ORM                      
┌─────────────────────────▼───────────────────────────────────┐
│                    SQLite  (inventory.db)                    │
│   Multi-Account · Products · Orders · Projects · Finance     │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 关键特性

| 特性 | 说明 |
|------|------|
| **多账本隔离** | 通过 `X-Account-ID` 请求头声明所属账本，所有 ORM 查询均带 `account_id` 过滤 |
| **命令模式** | 写操作封装为 `Command` / `CommandHandler` 对象，通过全局注册表分发 |
| **显式编排** | Command Handler 直接调用 `inventory_ops` / `income_ops` 完成库存和收入联动；EventBus 仅负责日志和汇总重算 |
| **领域模型** | 核心实体（采购单、销售单、项目成本）具备独立的业务规则验证层 |
| **订单类型** | `OrderType` 枚举区分零售(RETAIL)、项目(PROJECT)、人工采购(PURCHASE_LABOR)，显式控制库存和收入联动行为 |
| **库存可控** | `Product.track_inventory` 字段控制商品是否参与库存（人工类商品跳过） |
| **统一枚举** | 所有状态枚举定义于 `backend/enums.py`，通过 `/api/enums` 接口暴露 |
| **金额精度** | 使用 Python `Decimal` / 自定义 `Money` 值对象，统一 `round(2)` 防止浮点误差 |

### 1.2 技术栈总览

| 层次 | 技术 | 说明 |
|------|------|------|
| **前端框架** | Vue.js 3 + Vite | 组合式 API、单文件组件，Element Plus UI |
| **状态管理** | Pinia | 账本、枚举、项目全局状态 |
| **路由** | Vue Router 4 | 客户端 SPA 路由，20 个视图页面 |
| **HTTP 客户端** | Axios | 统一拦截器，注入 `X-Account-ID` / `X-Operator` |
| **后端框架** | FastAPI (Python 3.x) | 异步路由，Pydantic v2 数据校验 |
| **ORM** | SQLAlchemy 2.x | 同步 Session，自动迁移初始化 |
| **数据库** | SQLite | 单文件，适合单机/小团队部署 |
| **图片存储** | 本地文件系统 | `uploads/images/` 目录，防路径穿越校验 |
| **日志** | Python logging | 文件 + 控制台双输出，`app.log` |

## 3. 核心数据流程

### 3.1 销售单创建数据流

**销售单是系统中最复杂的业务流**，涉及库存、项目收入、项目汇总三条联动链路。

```
前端 Sales.vue                   后端 routers/sales.py
      │                                 │
      │─── POST /api/sales ─────────────▶│
      │    Body: SaleOrderCreate         │
      │                           create_sale_order()
      │                                 │
      │                      ┌───────────┴──────────────┐
      │                      │   Command: CreateSaleOrder │
      │                      │  1. 生成订单号 SO-日期-时分秒-N│
      │                      │  2. 校验商品不重复           │
      │                      │  3. 设置 order_type=RETAIL  │
      │                      │  4. 创建 SaleOrder ORM      │
      │                      │  5. 按行创建 SaleItem        │
      │                      │  6. _distribute_total_price  │
      │                      └───────────┬──────────────┘
      │                                 │
      │                       if status == COMPLETED:
      │                                 │
      │                      ┌───────────┴──────────────┐
      │                      │  显式编排（直接调用）：     │
      │                      │   ├─ inventory_ops         │
      │                      │   │   .sale_deduct() 扣库存 │
      │                      │   │   （零售+项目专属均扣） │
      │                      │   └─ income_ops            │
      │                      │       .sale_create_income() │
      │                      │  EventBus.emit()            │
      │                      │   └─ priority=10: 写操作日志 │
      │                      │   └─ priority=90: 项目汇总  │
      │                      └───────────┬──────────────┘
      │                                 │
      │                               db.commit()
      │◀── 200 SaleOrderOut ───────────┘
```

### 3.2 采购单数据流（库存入库）

```
POST /api/purchases
     │
dispatch(CreatePurchaseOrder, db)
     │
CreatePurchaseOrderHandler.handle()
     │
┌────▼──────────────────────────────────────────────────┐
│  1. 生成订单号 PO-{日期}-{时分秒}-{序号}                │
│  2. 校验商品不重复（Counter 查重）                       │
│  3. 设置 order_type=RETAIL                              │
│  4. 创建 PurchaseOrder + PurchaseItem                  │
│  5. if status=COMPLETED 且 product.track_inventory:     │
│       inventory_ops.purchase_add() → 库存 += quantity   │
│  6. order.total_price = Σ(item.total_price).round(2)    │
│  7. EventBus.emit() → priority=10: 写操作日志           │
└────────────────────────────────────────────────────────┘
     │
```

**状态变更时库存处理规则**（仅 `track_inventory=true` 的商品）:
- `PENDING` → `COMPLETED` : 加库存（所有行）
- `COMPLETED` → `CANCELLED` : 减库存（所有行）
- `CANCELLED` → `COMPLETED` : 加库存（所有行）
- 修改 items 时 : 旧行回补 → 删旧行 → 建新行 → 按新状态扣减

### 3.3 项目成本联动数据流

```
POST /api/project-costs
     │
dispatch(CreateProjectCost, db)
     │
CreateProjectCostHandler.handle()
     │
┌────▼────────────────────────────────────────────────────┐
│  1. 校验 project 存在 + cost_type 合法                    │
│  2. 创建 ProjectCost ORM                                  │
│  3. 显式编排联动（按 cost_type 分支）：                     │
│     ├─ 材料类:                                            │
│     │   ├─ _find_or_create_project_sale() 查找/创建项目销售单│
│     │   ├─ _upsert_sale_item() 写入销售明细                │
│     │   │   └─ CostItemLink 记录 cost→item 精确关联        │
│     │   └─ inventory_ops.cost_deduct_stock() 扣库存       │
│     └─ 人工类:                                            │
│         ├─ _find_or_create_labor_purchase() 查找/创建人工采购单│
│         └─ _upsert_purchase_item() 写入采购明细            │
│             └─ CostItemLink 记录 cost→item 精确关联        │
│  4. emit('project_cost.created') → priority=10: 写操作日志 │
│  5. update_project_summary() → 重算 Project 汇总           │
└─────────────────────────────────────────────────────────┘
```

### 3.4 财务数据流（期初余额 + 现金流水）

财务模块通过 Command 模式封装，所有操作均执行**资产负债平衡校验**（资产 = 负债 + 权益）后入库。

**期初余额创建流程**:
```
POST /api/opening-balances    
  → CreateOpeningBalance(Command)
  → CreateOpeningBalanceHandler
     ├─ 校验日期唯一性
     ├─ 校验 资产 = 负债 + 权益（会计恒等式）
     ├─ 创建 OpeningBalance ORM
     └─ 写操作日志
```

**现金流水三分类**:
- `operating` (经营活动) - 日常收支
- `investing` (投资活动) - 固定资产等
- `financing` (筹资活动) - 贷款/还款等

### 3.5 数据模型 ER 概览

```
Account (账本)
   │
   ├─── Product (商品) ────── Inventory (库存)
   │       │                     │
   │    PurchaseItem          SaleItem
   │       │                     │
   ├─── PurchaseOrder ───────SaleOrder
   │       │                     │
   │    [project_id]          [project_id]
   │         \                  /
   ├─────── Project ────────────
   │         │
   │    ┌────┴────────────┐
   │  ProjectCost    ProjectIncome
   │  (materials·人工·差旅)  (sales→income 联动)
   │       │
   │  CostItemLink (成本→行项精确关联)
   │
   ├─── Customer / Supplier (合作伙伴)
   ├─── Invoice (发票 进/销项)
   ├─── Expense (公司费用)
   ├─── PersonalTransaction (个人流水)
   ├─── OpeningBalance (期初余额)
   ├─── CashFlowTransaction (现金流水)
   └─── OperationLog (操作日志)
```

### 3.6 三大业务不变量

> 📌 **系统不变量（Invariants）**

| 序号 | 说明 |
|------|------|
| **不变量 I** | 库存一致性：`inventory.quantity = Σ(完成的采购行数量) − Σ(已完成销售单行数量, deduct_inventory=true) − Σ(项目材料成本数量)`；项目专属销售单和零售单均通过 `deduct_inventory=true` 扣减库存 |
| **不变量 II** | 项目收入一致性：`ProjectIncome` 记录与 `SaleOrder(project_id, status=completed)` 一一对应，通过 `UniqueConstraint(project_id, source_type, source_id)` 保证唯一 |
| **不变量 III** | 项目汇总一致性：`Project.total_cost` = 项目关联的已完成销售单中每个商品的进价(purchase_price)×数量求和；`Project.total_income` / `profit` 由 `update_project_summary()` 实时重算 |

## 4. 设计决策及理由

### 4.1 命令模式（Command Pattern）

| 维度 | 说明 |
|------|------|
| **决策** | 所有写操作封装为 `Command` dataclass + `CommandHandler`，通过 `dispatch()` 分发 |
| **理由** | 将业务意图与执行细节分离；单元测试可直接实例化 Command 对象注入 mock DB；命令注册表提供全局可见的操作清单 |
| **权衡** | 增加了少量模板代码（Command dataclass + Handler 类）；对于简单查询操作仍直接调用 CRUD 函数，避免过度工程化 |
| **现状** | 8 个命令模块，覆盖销售单、采购单、成本、收入、财务、发票、项目全部写操作；v4 重构后 Handler 显式编排库存和收入联动 |

### 4.2 事件总线（EventBus）

| 维度 | 说明 |
|------|------|
| **决策** | EventBus 仅承担日志记录和项目汇总重算两项职责；库存扣减、收入联动等核心业务副作用由 Command Handler 显式编排（`crud/inventory_ops.py` + `crud/income_ops.py`） |
| **理由** | v4 重构前 EventBus 承载了库存(p30)、业务(p50)、汇总(p90)三条回调链，调试困难且隐式耦合；重构后"谁创建，谁负责编排"，代码路径可读性大幅提升 |
| **权衡** | Command Handler 代码量增加（需显式调用 ops 函数）；但换来的是编排逻辑可见、可调试、可单元测试 |
| **现状** | EventBus 仅注册 2 类 Handler：priority=10（操作日志）、priority=90（项目汇总重算） |

### 4.3 多账本设计（Multi-Tenancy via Header）

| 维度 | 说明 |
|------|------|
| **决策** | 通过请求头 `X-Account-ID` 传递账本 ID，所有 ORM 查询均加 `account_id` 过滤 |
| **理由** | 轻量方案；无需 URL 路径中嵌入账本参数；适合小团队多账本（如日常+个人）场景 |
| **权衡** | 未实现行级安全；账本 ID 由客户端声明，需信任前端或后续增加鉴权层；无账本间数据共享机制 |

### 4.4 SQLite 单文件数据库

| 维度 | 说明 |
|------|------|
| **决策** | 使用 SQLite 作为唯一持久化存储 |
| **理由** | 零运维、易备份（单文件复制即可）；支持事务、外键、唯一约束；适合中小企业单机部署 |
| **权衡** | 写并发受限；不适合多节点横向扩展；数据量大时需迁移至 PostgreSQL |
| **现状** | 提供热备方案（`hot_backup.ps1`）；`database.py` 的 `init_db()` 在启动时自动完成 DDL 初始化和字段迁移 |

### 4.5 领域模型（Domain Model）

| 维度 | 说明 |
|------|------|
| **决策** | 核心实体（采购单、销售单、项目成本）具备独立 Domain Model 层，通过 `from_orm()` 从 ORM 对象构建 |
| **理由** | 业务规则（状态机、不变量校验）集中于 Domain 对象，不散落在 CRUD 中；便于单元测试（无需 DB）；Money 值对象统一金额精度 |
| **权衡** | Domain 层与 ORM 层并存，存在一定重复 |

### 4.6 前端枚举同步策略

枚举定义在后端 `backend/enums.py`（Single Source of Truth），通过 `GET /api/enums` 接口暴露给前端。前端 `stores/enums.js` 在应用启动时拉取并缓存，避免前后端枚举不一致。

### 4.7 金额精度保障

| 机制 | 实现方式 |
|------|----------|
| **Decimal 类型** | 所有金额计算使用 Python `Decimal`，避免浮点误差 |
| **Money 值对象** | `domain/money.py` 封装 Decimal，提供加减乘除和比较运算符重载 |
| **统一 round(2)** | 所有订单总价、报表金额统一 `.quantize(Q2)` 或 `round(2)` 保留2位小数 |
| **自定义总价** | `_distribute_total_price()` 将整单差额按比例分配到各行，确保行合计 = 订单总价 |

## 5. 系统限制和约束

### 5.1 基础设施限制

| 限制项 | 说明及影响 |
|--------|------------|
| **SQLite 写并发** | 支持多读单写；高并发写场景下存在锁竞争，不适合多人同时高频写入 |
| **单节点部署** | 无水平扩展能力；热备方案仅提供数据冷备，无主从复制 |
| **图片本地存储** | 业务图片存储在本地，不支持云存储；无法在多节点部署中共享文件 |
| **文件大小限制** | 图片上传限制 5MB，支持 jpg/png/gif/webp 格式 |

### 5.2 安全约束

| 约束项 | 说明 |
|--------|------|
| **无用户鉴权** | 系统未实现用户登录/权限体系；账本 ID 由客户端 Header 声明，依赖网络层隔离（局域网/本机部署） |
| **CORS 白名单** | 生产环境已限制为 localhost 白名单，可通过 `CORS_ORIGINS` 环境变量追加额外来源 |
| **路径穿越防护** | 图片删除操作通过 `os.path.realpath()` 校验确保文件路径在 `UPLOAD_DIR` 内 |
| **操作日志** | 所有写操作记录至 `OperationLog` 表，只写不删，提供审计追踪 |

### 5.3 业务逻辑约束

| 约束项 | 说明 |
|--------|------|
| **项目专属销售单必须扣库存** | `SaleOrder` 关联 `project_id` 且 `order_type=PROJECT` 时，`deduct_inventory` 必须为 `true`；零售单也必须为 `true`；仅 `deduct_inventory=true` 的销售单才扣减/回补库存 |
| **项目型销售单保护** | `order_type=PROJECT` 的销售单禁止删除、禁止修改 items、禁止切换项目，防止破坏项目成本链 |
| **人工采购单保护** | `order_type=PURCHASE_LABOR` 的采购单禁止删除/修改 items，由项目成本生命周期驱动 |
| **库存可控商品** | `Product.track_inventory=false` 的商品（如人工服务）跳过所有库存操作 |
| **订单内商品唯一** | 同一采购单/销售单内同一 `product_id` 只能出现一行；三层防护：CRUD 校验 + 前端防重 + 数据库 `UniqueConstraint` |
| **供应商/客户删除保护** | 有关联采购单/销售单的供应商或客户不可删除 |
| **商品删除保护** | 有采购或销售记录的商品不可删除 |
| **状态机约束** | 订单状态转换遵循：`PENDING`→`COMPLETED/CANCELLED`，`COMPLETED`→`CANCELLED`，`CANCELLED`→`COMPLETED`（恢复） |
| **资产负债平衡** | 期初余额创建/更新时强制校验：资产 = 负债 + 权益 |
| **项目收入唯一性** | `ProjectIncome` 表通过 `UniqueConstraint(project_id, source_type, source_id)` 防止重复记录 |

### 5.4 已修复陷阱（Anti-Pattern 红线）

> 📌 **以下问题已从代码中修复，但教训已固化到 `.joycode/rules/anti-patterns.md`，任何新改动必须先通过自检。**
>
> 13个具体陷阱由三个底层机制驱动：
> - **MR-1 先读后写**：AI 从空白开始写，不扫描项目已有防护/约定/数据源
> - **MR-2 捷径否决**：AI 优化"本次改动量最小"而非"全生命周期成本最小"
> - **MR-3 全局卫生**：AI 只追加不清理，把每次任务视为独立"添加"而非"系统演进"

| 编号 | 陷阱 | 根因 | 修复方式 | 元规则 |
|------|------|------|---------|--------|
| AP-1 | 迁移函数清空数据 | DELETE FROM 无幂等守卫 | 删除 `_migrate_v4_order_type()` | MR-2 |
| AP-2 | notes 字符串解析 | 用 notes 编码 cost_id 关联 | 建 `CostItemLink` 关联表 | MR-2 |
| AP-3 | EventBus 隐式联动 | condition=lambda 驱动核心业务 | Command Handler 显式编排 | MR-2 |
| AP-4 | toFixed 全局崩溃 | Decimal→JS 字符串，逐案修补 | `formatMoney()` + 全站扫描 | MR-1+MR-3 |
| AP-5 | 对账 API 404 | 前后端单复数不一致 | 统一为复数形式 | MR-1 |
| AP-6 | `||` 导致 0 值丢失 | JS falsy 陷阱 | 改用 `??` nullish coalescing | MR-1 |
| AP-7 | 重复 import/路由 | 不检查已有内容 | 添加前先 grep | MR-3 |
| AP-8 | 枚举硬编码 | 不知道全局枚举源 | 从 enumsStore 动态读取 | MR-1 |
| AP-9 | 两套体系共存 | 不删旧代码 | 立即删除或标记废弃 | MR-2+MR-3 |
| AP-10 | inline 渐变样式 | 不知道设计 token | 用 `--el-color-*` 变量 | MR-1+MR-2 |
| AP-11 | 副作用静默失败 | 空 catch/watch 泄漏/assert 校验 | catch 有反馈 + watch 绑定生命周期 + 禁止 assert | MR-1+MR-2 |
| AP-12 | ORM 跨账本泄漏 | primaryjoin 缺 account_id | JOIN 条件含 `account_id` | MR-1 |
| AP-13 | 前后端约定不对齐 | 容差/字段长度/组件兼容 | 精度对齐 + 长度留余量 + 兼容性验证 | MR-1 |

### 5.5 已知技术债务

> 📌 **技术债务清单**

| 序号 | 说明 |
|------|------|
| 1 | 部分业务规则散落在 CRUD 中而非 Domain 中 |
| 2 | 无 CI/CD 流水线；测试覆盖不完整 |
| 3 | 前端缺少全局错误处理和请求重试机制；部分页面数据未做缓存 |
| 4 | SQLite 数据库无在线迁移工具（如 Alembic） |
| 5 | 图片文件与业务数据不在同一事务中，业务回滚时可能出现孤儿文件 |

### 5.5 性能约束

| 问题 | 说明 |
|------|------|
| **N+1 查询** | 报表接口对每个项目实时调用 `update_project_summary()` |
| **无全文索引** | 库存列表的 search/category 筛选使用 LIKE 查询，数据量大时性能下降 |
| **响应体积** | 采购/销售列表接口返回全量 items 嵌套数组，大订单场景下单次响应体积较大 |
| **同步事件** | EventBus 为同步执行；若 Handler 耗时较长会阻塞请求响应 |

### 5.6 打包部署

| 机制 | 说明 |
|------|------|
| **PyInstaller 打包** | `inventory.spec` 配置，console=False（无CMD窗口），输出 `dist/进销存管理系统/` |
| **安装器** | `installer.py` tkinter GUI 安装向导，输出单文件 `进销存管理系统安装包.exe` |
| **端口检测** | `launcher.py` 启动时自动检测端口占用，从 8000 开始搜索至 8099，找到可用端口后启动 |
| **端口指定** | 环境变量 `INVENTORY_PORT` 可手动指定端口，优先级高于自动检测 |
| **端口记录** | 选定端口写入工作区 `port.txt`，供外部工具（安装器/快捷方式）读取 |
| **工作区隔离** | 打包模式下数据存 `%APPDATA%/进销存管理系统/`，与程序文件分离，卸载不删数据 |
| **崩溃日志** | console=False 模式下致命错误写入桌面 `进销存管理系统_启动失败.txt` + 工作区 `app.log` |
| **数据库模板** | 首次运行时从内置 `inventory.db.template` 复制空数据库到工作区 |
