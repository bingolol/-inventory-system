# 设计决策 (ADR)

> 架构决策记录 — 哪些看似"bug"的行为是故意设计的

---

### D-1: `_d(None)` 返回 `Decimal('0')` — 宁可算错不要崩

`utils._d()` 对 `None` 返回 `Decimal('0')`，不抛异常。

**为什么**：财务报表计算链很长（期初余额 → 损益 → 税额），如果某个中间值为 `None` 就崩，整个报表无法生成。返回 0 让计算继续，审计时可通过 `OperationLog` 追溯缺失字段。

**何时修复**：仅当业务明确要求"字段必填且必须阻止提交"时，应在 Schema 层校验，而不是在 `_d()` 层报错。

### D-2: SQLAlchemy 删除后访问属性不崩溃 — Python 对象有缓存

```python
order = db.query(SaleOrder).filter(...).first()
db.delete(order)
db.commit()
print(order.order_no)  # 不会报错，返回删除前的值
```

**为什么**：`db.commit()` 会 expire ORM 对象，但 Python 仍保留属性的缓存值。访问缓存值不会触发 lazy load，因此不会因行已删除而报 `ObjectDeletedError`。这是 SQLAlchemy 的正常行为。

**何时修复**：如果你看到代码依赖此行为，请确认不是偶然——理想情况下应在 commit 前保存需要的数据。

---

### D-3: `ConfirmStore` 用普通 dict — asyncio 单线程安全

`confirm_middleware.py` 的 `ConfirmStore` 用 `dict` 存储确认 token，没有加锁。

**为什么**：uvicorn 默认 asyncio 事件循环是单线程的，`dict` 在单线程下不会并发修改。加锁反而增加复杂度。

**何时修复**：如果部署改为多 worker（`--workers N`）或使用多线程服务器，需要改用 `threading.Lock` 或 Redis。

---

### D-4: `SaleOrderUpdate` 发送 `items: []` 会删除整张单 — 故意的破坏性行为

```python
PUT /api/sales/123
{"items": []}  # 看起来是"清空商品"，实际是"删除整张销售单"
```

**为什么**：销售单没有"空单"状态，商品行归零意味着单据无意义，自动删除避免僵尸数据。代码注释明确写了"商品行数归零自动删除"。

**何时修复**：如果用户反馈"不小心删了单"，可以改为需要显式 `DELETE /api/sales/123`，禁止通过 PUT 清空。

---

### D-5: `with_for_update()` 在 SQLite 下无效 — 当前单 worker 安全

`payments.py`、`receipts.py` 用 `.with_for_update()` 加行锁，但 SQLite 不支持 `SELECT FOR UPDATE`，这行代码等于没写。

**为什么**：代码预留了多数据库兼容性（未来可能切 PostgreSQL）。当前单 worker + SQLite 事务保证了串行执行，不会出竞态。

**何时修复**：部署多 worker 前必须改用乐观锁（version 字段）或切到 PostgreSQL。

---

### D-6: Schema 的 `model_validate` 篡改 ORM 对象 — 当前未被调用

`OpeningBalanceOut.model_validate()` 和 `FixedAssetOut.model_validate()` 会修改传入的 ORM 对象（`obj.date = obj.date.isoformat()`）。

**为什么**：这两个 `model_validate` 目前在代码库中从未被调用。路由层用的是手动构造 `_to_out()`，不触发此问题。

**何时修复**：如果未来需要调用 `model_validate`，应改为不修改原对象（用 `copy` 或在序列化时转换）。

---

### D-7: 删除后访问已删除对象 — 同 D-2

`sales.py` 和 `purchases.py` 的删除路由在 `unit_of_work` 提交后访问 `order.order_no`。同 D-2，这是安全的。

### D-8: 冲红凭证日期必须用原 AccountMove.date（BR-22，避免 off-by-one）

`finance_integration.py` 的 `reverse_journal()` 默认用 `original.date`（原凭证日期），**不**用 `date.today()`。`engine_inventory.py` 的 `InventoryEngine.reverse()` 的 StockMove 反向流水 `move_date` 也是从原单据取业务日期（通过 `_get_move_date(source_type, source_id)`）。

**为什么**：

- `AccountMove.date` 字段类型是 `Date`（不是 `DateTime`），只精确到日，无时间分量。
- BS 报表按 `date <= 'YYYY-MM-DD'` cutoff 过滤，只取截止日期前的凭证。
- 如果反向凭证用今天的日期：
  - **资产侧**：StockMove 反向流水 `move_date` 取原单日期 → 进 cutoff 内 → 库存值 +冲回金额
  - **权益侧**：AccountMove 反向凭证 `date` 是今天 → 不进 cutoff → 1405 ledger 仍按"已销售"状态计算
  - **结果**：`A - (L + E) = X` 不平（典型 diff = 一笔销售的不含税金额，如 40000）

**机理**：StockMove 反向流水与 AccountMove 反向凭证是同一笔业务的两面，必须用同一日期，否则 cutoff 过滤会漏掉一面。

**相关代码**：
- `finance_integration.py:194`: `date=reversal_date or original.date`
- `engine_fixed_asset.py:163`: `disposal_date = disposal_date or date.today()`（同类问题，处置也允许传 disposal_date）

**何时不可修复**：此为硬约束，禁止改回 `date.today()`。详见 `project_memory.md` BR-22。

### D-9: 退货成本用原始 StockMove.unit_cost（移动加权平均法历史快照）

`sale_commands.py` 的 `ReturnSaleOrderHandler` 在计算销售退货 COGS 冲回时，从**原销售时锁定的 `StockMove.unit_cost`** 取值，**不**用当前 `Inventory.average_cost`。

**为什么**：

- 移动加权平均法下，每次销售时锁定的 `unit_cost` 是历史快照，反映当时出库的移动加权平均成本。
- 退货应按当时锁定的成本反向冲回，保持"卖出 + 退货"COGS 净影响为 0。
- 如果用当前 `Inventory.average_cost` 冲回：
  - 期间若发生新采购（average_cost 上升）或新销售（average_cost 可能变化），冲回金额会与原销售时锁定金额不一致，产生"退货差异"。
  - 这个差异在会计上无意义（不是真实的损益），却会污染利润表。

**机理**：移动加权平均法要求"先进先出"的成本与"反向冲回"的成本配对。原 StockMove 是当时成本事实的真相源，`Inventory.average_cost` 只是当前缓存。

**相关代码**：
- `sale_commands.py:302-308`:
  ```python
  move = db.query(StockMove).filter(
      StockMove.source_type == "sale_order",
      StockMove.source_id == order.id,
      StockMove.product_id == pid,
  ).first()
  unit_cost = move.unit_cost if move and move.unit_cost else Decimal("0")
  cost_return += (qty_ret * unit_cost).quantize(Q2)
  ```
- 同原则适用于采购退货：`ReturnPurchaseOrderHandler` 也从原 StockMove 取 `unit_cost`。

**何时修复**：此为硬约束，禁止改用 `Inventory.average_cost`。如果 `StockMove.unit_cost` 缺失（早期数据），回退到 `Inventory.average_cost` 是可接受的兜底，但新数据必须保证 `StockMove.unit_cost` 已锁定。

### L-1: 跨年补提所得税未区分场景

**现象**：月结计提所得税时（`engine_tax.py` 调用 `AccountingEngine.calculate_income_tax()`），无论业务日期是否跨年，都按当期所得税科目计提，未使用"以前年度损益调整"科目。

**正确做法**：
- 当期补提当期所得税：`借: 6801 所得税费用 / 贷: 222105 应交企业所得税`（已实现）
- 跨年补提以前年度所得税：应使用 `借: 6901 以前年度损益调整 / 贷: 222105 应交企业所得税`，并结转至未分配利润 `借: 3004 利润分配-未分配利润 / 贷: 6901 以前年度损益调整`

**当前状态**：月结引擎未区分跨年场景，统一按当期所得税费用（6801）计提。这是原有设计局限，本次修复（D-8 BR-22）未改变日期逻辑。

**影响**：跨年补提所得税会虚增当期所得税费用，影响利润表准确性。

**待开发**：在 `engine_tax.py` 的 `execute()` 中添加 `is_prior_year` 参数判定，业务日期所在年 ≠ 当前会计年度时，改用 6901 以前年度损益调整科目过账。

**优先级**：中（仅跨年场景受影响，当年内月结不受影响）

### L-2: 1901 待处理财产损溢转销缺失

**现象**：库存调整（`product_commands.py` 的 `AdjustInventoryHandler`）在盘盈/盘亏时直接生成凭证：
- 盘亏：`借: 6601 管理费用 / 贷: 1405 库存`
- 盘盈：`借: 1405 库存 / 贷: 6601 管理费用`

跳过了 `1901 待处理财产损溢` 科目的过渡。

**正确做法**（企业会计准则）：
1. **审批前**（发现盘盈/盘亏）：
   - 盘亏：`借: 1901 待处理财产损溢 / 贷: 1405 库存`
   - 盘盈：`借: 1405 库存 / 贷: 1901 待处理财产损溢`
2. **查明原因后转销**：
   - 盘亏（管理责任/正常损耗）：`借: 6601 管理费用 / 贷: 1901`
   - 盘亏（责任人/保险公司赔偿）：`借: 1221 其他应收款 / 贷: 1901`
   - 盘亏（非常损失）：`借: 6711 营业外支出 / 贷: 1901`
   - 盘盈：`借: 1901 / 贷: 6601 管理费用`（或 `贷: 6301 营业外收入`，视性质而定）

**当前状态**：系统未实现 1901 自动转销流程，直接一步生成损益凭证。需人工补充审批凭证。

**影响**：缺少"待批准"中间状态，无法区分"已发现未审批"和"已审批转销"两个阶段。

**待开发**：
1. 在 `AdjustInventory` 命令添加 `approved: bool` 字段（默认 False）
2. `approved=False` 时生成 1901 过渡凭证（不直接影响损益）
3. 新增 `POST /api/inventory/{id}/approve` 端点，将 1901 余额转销至损益科目
4. 凭证 builder 添加 `inventory_loss_approved` 和 `inventory_gain_approved` 类型

**优先级**：中（小企业会计准则允许简化处理，但大中型企业需要此流程）
