# 设计决策 (ADR)

> 架构决策记录 — 哪些看似"bug"的行为是故意设计的

---

### D-1: `_d(None)` 返回 `Decimal('0')` — 宁可算错不要崩

`utils._d()` 对 `None` 返回 `Decimal('0')`，不抛异常。

**为什么**：财务报表计算链很长（期初余额 → 损益 → 税额），如果某个中间值为 `None` 就崩，整个报表无法生成。返回 0 让计算继续，审计时可通过 `OperationLog` 追溯缺失字段。

**何时修复**：仅当业务明确要求"字段必填且必须阻止提交"时，应在 Schema 层校验，而不是在 `_d()` 层报错。

### D-2: SQLAlchemy 删除后访问属性不崩溃 — Python 对象有缓存

```python
order = db.query(SaleOrder).filter(...).first()
db.delete(order)
db.commit()
print(order.order_no)  # 不会报错，返回删除前的值
```

**为什么**：`db.commit()` 会 expire ORM 对象，但 Python 仍保留属性的缓存值。访问缓存值不会触发 lazy load，因此不会因行已删除而报 `ObjectDeletedError`。这是 SQLAlchemy 的正常行为。

**何时修复**：如果你看到代码依赖此行为，请确认不是偶然——理想情况下应在 commit 前保存需要的数据。

### D-3: `ConfirmStore` 用普通 dict — asyncio 单线程安全

`confirm_middleware.py` 的 `ConfirmStore` 用 `dict` 存储确认 token，没有加锁。

**为什么**：uvicorn 默认 asyncio 事件循环是单线程的，`dict` 在单线程下不会并发修改。加锁反而增加复杂度。

**何时修复**：如果部署改为多 worker（`--workers N`）或使用多线程服务器，需要改用 `threading.Lock` 或 Redis。

---

### D-4: `SaleOrderUpdate` 发送 `items: []` 会删除整张单 — 故意的破坏性行为

```python
PUT /api/sales/123
{"items": []}  # 看起来是"清空商品"，实际是"删除整张销售单"
```

**为什么**：销售单没有"空单"状态，商品行归零意味着单据无意义，自动删除避免僵尸数据。代码注释明确写了"商品行数归零自动删除"。

**何时修复**：如果用户反馈"不小心删了单"，可以改为需要显式 `DELETE /api/sales/123`，禁止通过 PUT 清空。

### D-5: `with_for_update()` 在 SQLite 下无效 — 当前单 worker 安全

`payments.py`、`receipts.py` 用 `.with_for_update()` 加行锁，但 SQLite 不支持 `SELECT FOR UPDATE`，这行代码等于没写。

**为什么**：代码预留了多数据库兼容性（未来可能切 PostgreSQL）。当前单 worker + SQLite 事务保证了串行执行，不会出竞态。

**何时修复**：部署多 worker 前必须改用乐观锁（version 字段）或切到 PostgreSQL。

---

### D-6: Schema 的 `model_validate` 篡改 ORM 对象 — 当前未被调用

`OpeningBalanceOut.model_validate()` 和 `FixedAssetOut.model_validate()` 会修改传入的 ORM 对象（`obj.date = obj.date.isoformat()`）。

**为什么**：这两个 `model_validate` 目前在代码库中从未被调用。路由层用的是手动构造 `_to_out()`，不触发此问题。

**何时修复**：如果未来需要调用 `model_validate`，应改为不修改原对象（用 `copy` 或在序列化时转换）。

---

### D-7: 删除后访问已删除对象 — 同 D-2

`sales.py` 和 `purchases.py` 的删除路由在 `unit_of_work` 提交后访问 `order.order_no`。同 D-2，这是安全的。
