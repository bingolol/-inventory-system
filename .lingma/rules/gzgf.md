# 强制工作流程（必须遵守）

## 必读文档

`docs/架构参考.md` 是本项目的架构权威参考，包含数据流、三大不变量、7条业务约束、设计决策及技术债务。**每个会话只要涉及本项目，必须在架构探查阶段阅读该文档中与当前任务相关的章节**，作为方案和审查的校验依据。不得凭记忆或臆断处理不变量和业务约束。

### 项目文档体系

| 文档 | 用途 |
|------|------|
| `docs/架构参考.md` | 架构权威参考（数据流、不变量、约束、设计决策） |
| `docs/开发速查表.md` | 日常开发快速参考（启动流程、文件索引、常见问题） |
| `docs/文件索引.md` | 按功能分类的完整文件列表 |

每个待办事项必须严格按以下流程执行，不可跳过任何步骤：

1. **架构探查**：动手写方案之前，先读现有代码，理解以下要点：
   - **分层定位**：改动涉及哪几层（router → command → domain → crud → model）？同层有没有可参照的现有实现？
   - **事件联动**：改动是否触发事件？是否影响库存扣减/收入生成/汇总重算等联动链路？
   - **复用机会**：现有工具函数、基类、composable、schema 是否已有可复用的？
   - **命名风格**：现有同类功能的命名、文件组织方式是什么？
   - **不变量校验**：改动是否触及三大不变量？（I-库存一致性、II-项目收入一致性、III-项目汇总一致性）若触及，必须在方案中说明如何保证不变量不被破坏
   - **约束合规**：改动是否涉及文档5.3节的7条业务逻辑约束？（项目销售禁扣库存、订单内商品唯一、删除保护、状态机、资产负债平衡等）方案必须声明合规
   - 必须引用具体的文件名和代码段作为探查依据，禁止凭空想象

2. **写方案**：基于架构探查结果，输出结构化方案，必须包含：
   - 涉及哪些文件、改什么、怎么改（具体到函数/类级别）
   - 如何融入现有架构模式（如 Command 模式、Domain 不变量校验、事件驱动联动）
   - 复用了哪些现有代码，为什么这是最优雅的路径
   - 副作用分析：改动会触发哪些联动，是否需要调整事件 handler
   - 不变量声明：列出受影响的不变量及保障措施
   - 约束合规声明：逐一检查7条业务约束，声明合规或说明例外

3. **审查与修正**：对方案进行四维度审查，发现问题则修正后重新审查，迭代直到通过：
   - **语法检查**：语法正确、类型匹配、导入完整
   - **逻辑检查**：业务逻辑正确、边界情况覆盖、异常处理完备
   - **优雅度审查**：是否有更简洁的写法？是否引入了不必要的复杂度？是否与现有代码风格一致？能否复用已有代码而非重复实现？
   - **不变量审查**：三大不变量是否仍成立？7条业务约束是否被遵守？

4. **执行修改**：按审查通过的方案改代码

5. **测试验证**：对更新过的功能和模块进行实际测试，确保100%落地无bug

6. **更新 repomix 代码快照**：一批相关任务完成后，在项目根目录执行 `npx repomix` 更新 repomix-output.md，确保 AI 上下文中的代码快照与最新代码一致

7. **同步更新项目文档**：根据改动范围更新对应文档
   - **必更**：如果改动涉及架构参考中已有内容（数据流、不变量、约束、技术债务、API端点、枚举值等），更新 `docs/架构参考.md`
   - **必更**：如果改动新增/删除/重命名了文件，更新 `docs/文件索引.md`
   - **条件更**：如果文件索引有变动，同步更新 `docs/开发速查表.md` 中的文件索引章节
   - **条件更**：如果新增了常见问题或排障经验，补充到 `docs/开发速查表.md`

8. **写入更新日志**：每批工作完成后，必须将变更内容追加写入 `UPDATE_LOG.md`
   - **必须包含绝对时间戳**：条目标题格式为 `## YYYY-MM-DD HH:MM 变更摘要`（如 `## 2026-05-03 14:27 销售单增强`）
   - 每条变更列出：改了什么、涉及哪些文件（具体到函数/类级别）、关键设计决策
   - 新条目追加到文件顶部（最新在前）

## 项目架构参考


后端分层：router → command → domain → crud/repository → model
联动机制：events.py 事件总线 + handlers.py 事件处理器
前端分层：views → composables → api/，stores 管理全局状态


## 禁止事项

- 不做架构探查直接写方案
- 不写方案直接改代码
- 改完不测试就跳到下一个待办
- 重复实现已有功能而不复用现有代码
- 触及不变量/业务约束的改动未在方案中声明影响和保障措施
